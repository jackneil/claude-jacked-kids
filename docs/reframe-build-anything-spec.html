Reframe: from "games" to "build anything" 🎮 → 🛠️

Project: claude-jacked-kidsDate: 2026-06-28Status: ready for /goal
Vision. A warm, kid-safe building buddy for children ~5–14. A kid describes anything they want — a game, or any app/tool/toy they dream up (a movie-release countdown, a chore chart, a flashcard quiz, a drawing toy, a tracker, a clock, a "rate my sandwich" picker — whatever random thing) — and the buddy builds the whole thing, shows it on screen, teaches a little along the way, and keeps it safe. Not games-only.
Scope discipline (the lesson that shaped this spec). We are not building feature-specific skills (no "pull my sports stats" skill, no validating a kid's idea). The skills stay generic: they help a kid build whatever they ask for and teach them as they go. Examples in this doc (movie countdown, stats tracker, etc.) are illustrations of the range, never features to implement individually.

1. Principles

  • Additive — zero breakage. Keep build-my-arcade and every internal symbol (window.GAMES, games/<id>/, GAME_META, ArcadeScores, the skill names). Add apps as a parallel structure next to games. Reframe only the kid-facing words.
  • "App" = any non-game creation (tool, tracker, dashboard, toy, list, quiz…). One generic make-an-app skill builds them all from the kid's plain-words request.
  • Teach along the way. The existing teach-and-check Brain-Boost gate works for apps too (its concepts — variable, loop, save, list — are universal); add a few app-flavored concept cards.
  • Safety unchanged, applied to apps. Local-first; first-name only; never store/publish last name, address, school, phone, photos; under-13 needs a grown-up to publish; scrub identifying info before publishing. If a creation holds personal info, it stays local and isn't published with identifying details. (No domain-specific machinery — the existing identity rules cover it.)
  • Per-project CLAUDE.md (the owner's idea). The buddy writes a lean CLAUDE.md into the kid's creations folder so the skills cohere — and so even a fresh chat (or a friend's plain Claude Code with no plugin) honors the warm tone, the folder conventions, which skill to use when, and the privacy floor.

3. Apps as a parallel structure

Mirror the proven game-island pattern so nothing about games changes:

<arcade>/
  index.html            # hub — now lists a "Games" zone AND a "My Apps & Tools" zone
  CLAUDE.md             # NEW: the project file the buddy writes (tone + rules + which skill when)
  assets/
    games-manifest.js   # window.GAMES  (unchanged)
    apps-manifest.js    # window.APPS   (new, peer)
    scores.js           # window.ArcadeScores (unchanged)
    store.js            # window.AppStore — tiny localStorage helper for app data (new, peer)
  games/<id>/           # unchanged
  apps/<id>/            # new, peer: index.html + app.js + metadata.js (window.APP_META)
  .jacked-kids/         # Player Card etc. — local-only, gitignored (unchanged)

A pre-existing (games-only) hub is idempotently retrofitted on first app: inject the apps render zone, create an empty apps-manifest.js + store.js + apps/, add the CLAUDE.md if missing. Offline-first rules are unchanged (script-globals not fetch, opens from a file; the dev/play server for anything that needs serving).

4. The per-project CLAUDE.md

Written/updated by build-my-arcade and the make/change skills into <arcade>/CLAUDE.md. Lean and self-honest — no routing table that can rot, no impossible self-updating. It contains:

  • Tone: talk like the warm kid buddy (so even a no-plugin Claude Code behaves).
  • What's here: a short, derived list of the kid's games + apps and the folder layout.
  • Which skill when: plain pointers ("to make a new thing → make-a-game / make-an-app; to change one → change-it; to show it → play-it").
  • The safety floor: local-first; first-name only / no PII; a grown-up publishes. Stated inline so it can't go stale by naming a skill that might change.
  • Merge, never clobber an existing CLAUDE.md.

5. Safety & privacy (existing rules, now covering apps)

No new machinery — just make sure the established rules apply to apps the same as games:

  • Everything runs locally; nothing about the kid leaves the device unless a grown-up publishes.
  • First name max — never last name, address, school, phone, email, photos — in anything stored or shared.
  • An app that holds personal info (a tracker, a journal) stays local; publishing is a grown-up step (under-13) and scrubs identifying details first (a handle/nickname + the content, never identity).
  • .jacked-kids/ stays gitignored and out of every publish path.

7. Explicitly cut (was over-engineering)

Dropped from the first design pass because it over-literaled one example: the sports-stats data ladder, ESPN/league snapshots, roster-CSV import, the two-zone identity system, the allowlist publish-scrubber, and any feature-specific stats skill. Generic "build the thing + existing safety + local-first" covers the real intent. If a kid wants to pull data into a creation, the generic make-an-app helps them figure it out in the moment — no bespoke skill.

8. Risks

  • file:// storage quirks (Safari) — bias to the play-server + visible save failures + export/import; verify on Safari in M7.
  • CLAUDE.md staleness — keep it lean + inline-safety; the build step regenerates the derived part.
  • Folder-sharing bypasses git/deploy scrubs — documented residual risk; the for-grownups note is the mitigation.
  • Scope creep — keep skills generic; resist building feature-specific helpers.
